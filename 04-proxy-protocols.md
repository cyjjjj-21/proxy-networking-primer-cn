# 代理协议：SOCKS5、Shadowsocks、VLESS、Reality

## 正向代理和反向代理

代理的核心含义是：一方不直接和目标通信，而是让中间节点代为转发。但“代理站在哪一边”不同，会分成正向代理和反向代理。

### 正向代理

正向代理站在客户端一侧，替用户访问外部服务。

```text
用户设备
-> 正向代理
-> 目标网站
```

目标网站看到的是代理出口 IP，而不是用户本地出口 IP。

常见用途：

```text
开发调试
企业内网出口
隐私保护
网络加速
按规则分流
链式代理
```

本知识库里的 Clash Verge、Shadowrocket、Xray 入口、SOCKS5 落地代理，主要讨论的都是正向代理体系。

### 反向代理

反向代理站在服务器一侧，替后端服务接收用户请求。

```text
用户浏览器
-> 反向代理
-> 后端网站 / API 服务
```

用户通常不知道后面真正有几台服务器，也不直接连接后端服务。

常见反向代理：

```text
Nginx
Caddy
HAProxy
Traefik
Cloudflare
CDN 边缘节点
API Gateway
负载均衡器
```

反向代理常见用途：

```text
TLS 证书终止
负载均衡
隐藏后端真实地址
缓存静态资源
WAF / 安全过滤
限流
灰度发布
统一 API 入口
```

### 两者的根本区别

```text
正向代理:
保护或代表客户端。目标网站看到代理。

反向代理:
保护或代表服务器。用户看到代理。
```

一个生活比喻：

```text
正向代理:
用户找代购去买东西，商家看到的是代购。

反向代理:
公司前台替内部员工接待访客，访客看到的是前台。
```

在网络安全和隐私语境里，正向代理常被用来改变用户出口；在网站架构里，反向代理常被用来承载入口、保护后端和分配流量。

### 反向代理和 Token 骗局

反向代理本身是正常技术。企业把 API 服务放在反向代理后面，用它做统一入口、鉴权、限流、日志、缓存和负载均衡，都是常规架构。

风险出现在“不可信反向代理”伪装成某个 AI 服务、API 中转站、加速站或镜像站时。用户以为自己在访问原始服务，实际请求先进入第三方代理：

```text
用户
-> 不可信反向代理
-> 原始 AI / API 服务
```

如果用户在这个代理页面登录、填写 API Key、上传文件、提交代码或输入敏感 prompt，代理方可能看到：

```text
登录态 Cookie
API Key
请求正文
上传文件
返回内容
调用频率
账号标识
```

这里要区分两种常被混淆的 Token：

```text
计费 token:
大模型按输入和输出文本量计费的单位。

认证 token:
API Key、Session Cookie、访问令牌等身份凭证。
```

所谓“通过反向代理大量获取 token，再做中转服务获利”，很多时候不是在合法获得计费 token，而是在诱导用户把认证凭证、登录态或请求内容交给第三方代理。后续可能变成账号池滥用、额度盗刷、隐私泄露、违规转售或钓鱼。

防骗原则：

```text
不要把 API Key 填进来源不明的中转站
不要在仿冒页面登录重要账号
不要把浏览器 Cookie / Session 复制给别人
不要把重要代码、文件、客户数据交给陌生代理
给 API Key 设置额度、权限和过期时间
发现异常调用后立即吊销密钥
优先使用官方 API、可信企业网关或自建网关
```

最关键的判断是：TLS 连接在哪里终止，谁就能看到那一段解密后的请求。浏览器到陌生中转站是 HTTPS，不代表陌生中转站到原始服务之间发生了什么也对用户透明。

### 为什么第三方反代更像“骗用户信任”，正向代理更像“隐藏用户来源”

正常 HTTPS 体系下，第三方很难把公认官方域名偷偷反代到自己的服务器。用户访问 `https://example.com` 时，浏览器会检查：

```text
地址栏域名是否是 example.com
TLS 证书是否属于 example.com
证书是否由可信 CA 签发
```

如果第三方没有官方域名的证书，即使 DNS 或网络链路把连接导到别的服务器，浏览器也会报证书错误。除非出现恶意根证书、企业级受信任中间人、系统被攻破、CA 事故等特殊情况，普通第三方不能无声替换官方域名。

常见骗局通常不是“把官方域名偷偷换掉”，而是诱导用户访问另一个域名：

```text
example0.com
example-mirror.net
example-api-proxy.example
```

这个域名可能页面长得像原站，也可能在背后反代原站。浏览器没有隐瞒地址栏；风险来自用户误信了非官方入口。

浏览器里的“源”也很重要：

```text
源 = 协议 + 域名 + 端口
```

`https://example.com` 和 `https://example0.com` 是两个不同的源。后者不能自动读取前者的 Cookie。它通常只能通过仿冒页面、低价中转、社群背书、复制 Cookie/API Key 等方式诱导用户主动交出凭证。

所以第三方反代的防骗重点是：

```text
看清域名
看清证书
不要在非官方域名登录重要账号
不要把 Cookie、API Key、文件交给陌生中转
```

正向代理的信息差不同：

```text
用户 -> 正向代理 -> 目标服务器
```

目标服务器收到的是代理出口 IP。它可能通过 IP 库、ASN、行为、指纹、频率、账号关系判断这是代理、机房、住宅代理或高风险来源，但它通常不能从一次普通请求里直接知道用户真实本地 IP。

一句话：

```text
第三方反代主要骗用户信任一个非官方入口。
正向代理主要让服务器看不到真实用户来源。
```

## 代理协议怎么判断新旧和能力

不要简单按“新就是好、旧就是差”判断。看这些维度：

```text
适合做公网入口吗
有没有加密
有没有伪装能力
能否承载复杂路由
客户端生态是否成熟
服务商是否提供这种接口
```

## HTTP 代理

老牌协议，主要用于 HTTP/HTTPS。

```text
优点: 简单，很多软件支持
缺点: 通用性有限，不适合现代强伪装公网入口
```

## SOCKS5

SOCKS5 是通用代理协议，不关心上层是网页、SSH、下载还是别的 TCP 流量。

它做的是：

```text
客户端告诉 SOCKS5 服务器:
请你帮我连接 example.com:443

SOCKS5 服务器:
替客户端连接目标，并在两边之间转发数据
```

特点：

```text
通用
轻量
支持用户名密码
适合服务商落地代理接口
本身不强调加密或伪装
```

静态 ISP 代理通常提供 SOCKS5，所以 Xray 出站可以用 `socks` 接它。

## Shadowsocks

Shadowsocks 是轻量加密代理协议。

```text
SOCKS5:
通用代理接口，本身不强调加密

Shadowsocks:
代理协议 + 客户端到服务端加密
```

它配置简单、速度好、生态成熟，但单独使用时伪装能力有限，复杂路由也不如 Xray/VLESS 体系灵活。

Shadowsocks 更像“一体化轻量代理”：

```text
代理请求
加密方法
共享密码
客户端到服务端转发
```

这些核心能力被做成一套简单方案。它不是粗糙，而是工程取向更轻、更少配置、更容易部署。

## VMess、Trojan、Hysteria、TUIC

```text
VMess:
曾经常用，设计较老，新部署通常不优先。

Trojan:
形态接近 HTTPS，通常依赖域名和证书。

Hysteria / TUIC:
偏 UDP/QUIC、高性能、弱网优化，依赖 UDP 网络环境。
```

## VLESS

VLESS 是 Xray 支持的代理协议。更精确地说，它在协议栈里承担“代理请求语义层”的职责。它告诉服务端：

```text
我是哪个用户
我要连接哪个目标地址
目标端口是多少
这是 TCP 还是 UDP
```

所以 VLESS 和 Shadowsocks 在大类上是平级的，都是代理协议。但设计取向不同：

```text
Shadowsocks:
轻量、一体化、配置少，代理和加密方案绑定得更紧。

VLESS:
模块化、分层更细，代理语义和传输安全分开组合。
```

常见字段：

```text
server
port
uuid
flow
tls/reality
sni
public-key
short-id
```

VLESS 通常不裸奔，常和传输/安全层组合：

```text
VLESS + TCP + Reality
VLESS + TCP + TLS
VLESS + WebSocket + TLS
VLESS + gRPC + TLS
```

示例组合：

```text
VLESS + TCP + Reality + xtls-rprx-vision
```

这里的分工是：

```text
VLESS:
表达代理请求，我是谁、我要连哪里。

TCP / WebSocket / gRPC:
负责传输形态，字节怎么送过去。

TLS / Reality:
负责安全、认证、加密和伪装。

Xray routing:
负责根据用户、域名、IP 等条件决定走哪个出口。
```

不同 UUID 还能作为路由身份：

```text
桌面端 UUID -> 默认走静态 ISP 代理
移动端 UUID -> 命中 direct 路由
```

可以用一个比喻记：

```text
Shadowsocks:
一把轻便折叠刀，少配置、好携带、够用。

VLESS + Reality:
模块化工具箱，组件多、配置复杂、可定制能力强。
```

## Reality

Reality 是 Xray 的 TLS 伪装与认证方案。它解决：

```text
客户端和服务端安全认证、加密通信
外部观察时更像普通 TLS 连接
不强依赖自有域名证书
```

字段：

```text
服务端私钥字段:
只放 VPS

public-key:
客户端使用

short-id:
客户端服务端匹配

sni / serverName:
外观看起来访问的 TLS 域名

dest:
服务端伪装回落目标

client-fingerprint:
模拟浏览器 TLS 指纹
```

Reality 不会让“用户连接了 VPS IP”这件事消失。它保护的是连接内容、代理协议细节、客户端认证和外部流量形态。

## 为什么示例前半段用 VLESS，后半段用 SOCKS5

链路：

```text
桌面端 / Clash Verge
-> VLESS Reality
-> 入口 VPS / Xray
-> SOCKS5
-> 静态 ISP 代理
-> 目标网站
```

前半段 `桌面端 -> VPS` 是公网入口，需要加密、认证、伪装、多用户区分和路由能力，所以用 VLESS Reality。

后半段 `VPS -> 静态 ISP 代理` 是服务商提供的落地代理接口。如果服务商给的是 SOCKS5 地址、端口、用户名、密码，Xray 出站就接 SOCKS5。

结论：

```text
VLESS Reality = 适合做安全公网入口
SOCKS5 = 适合做服务商落地出口接口
```

如果下一跳是自建落地 Xray，也可以用 VLESS/Trojan/Shadowsocks 等协议连接。
