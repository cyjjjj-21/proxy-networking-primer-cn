# 泄漏与信任模型：谁能看到什么

## 代理不是隐身斗篷

代理改变的是目标网站看到的出口，以及本地网络能看到的内容范围。它不会让所有网络元数据消失。

直接访问：

```text
用户设备 -> 目标网站
```

使用 VPS：

```text
用户设备 -> VPS -> 目标网站
```

链式代理：

```text
用户设备 -> VPS -> 静态 ISP 代理 -> 目标网站
```

## 本地运营商能看到什么

可能看到：

```text
用户真实 IP
用户连接了哪个 VPS IP
目标端口
连接时间
流量大小
持续时间
```

正常看不到：

```text
代理内部请求
最终访问的具体网页内容
HTTPS 明文内容
```

## 目标网站能看到什么

通常看到：

```text
出口 IP
浏览器指纹
Cookie
账号行为
请求频率
语言、时区、分辨率
```

通常看不到：

```text
用户家庭宽带真实 IP
用户设备到 VPS 的连接链路
```

除非存在 DNS、WebRTC、IPv6、设备 SDK、账号资料、支付信息等额外关联。

## 不可信中转能看到什么

任何代理、中转、反向代理或 API 网关，只要处在请求链路中间，都应被视为信任点。它可能只是转发流量，也可能记录请求。

如果使用不可信的 AI/API 中转站、镜像站或“加速站”，风险不只是不稳定，还包括凭证泄露：

```text
API Key
登录态 Cookie
请求正文
上传文件
返回内容
账号标识
调用频率
```

反向代理和普通正向代理的方向不同，但信任问题类似：谁接收了解密后的请求，谁就有机会读取、记录、转发或篡改这些内容。

第三方反向代理通常骗的不是浏览器的 TLS 验证，而是用户对入口域名的信任。官方域名和第三方镜像域名是不同的源；第三方镜像不能天然拿到官方站的 Cookie，除非用户主动登录、复制凭证、上传文件或填写 API Key。

正向代理的风险方向相反。目标服务器面对代理出口时，通常不知道真实用户本地 IP，只能根据出口 IP、ASN、设备指纹、账号行为和请求模式做概率判断。

安全做法：

```text
重要账号只在官方域名登录
API Key 只交给官方 SDK、可信企业网关或自建服务
敏感文件和代码不要经过陌生中转
为 Key 设置最小权限、额度和有效期
定期查看调用日志，异常时立即吊销
```

## VPS 是核心信任点

VPS 服务商或被完全控制的 VPS 可能看到：

```text
谁连接进 VPS
连接来源 IP
连接时间和流量大小
VPS 又连接了哪些外部 IP
DNS 查询记录，如果 DNS 由 VPS 处理
```

如果目标网站使用 HTTPS，VPS 通常看不到网页正文、账号密码、表单内容。但它仍能看到或推断元数据。

## 静态 ISP 代理 / 落地代理能看到什么

可能看到：

```text
VPS 在连接它
最终访问的目标 IP
连接时间
流量大小
```

目标网站看到的是 静态 ISP 代理 出口 IP。

## 强权限调查方

普通目标网站通常查不到谁在 VPS 背后。但强权限、多方日志配合时，理论上可以关联：

```text
目标网站日志 -> VPS/静态 ISP 代理 出口
VPS 服务商日志 -> 谁连入 VPS
本地运营商日志 -> 真实用户连接 VPS
```

核心结论：

```text
加密保护内容
代理改变出口
DNS 影响解析和画像
VPS 是信任点
落地 IP 决定目标网站看到的身份
路由元数据不会凭空消失
```

## IPv6 泄漏

危险状态：

```text
IPv4 = 代理出口
IPv6 = 本地运营商
```

好状态：

```text
IPv6 not reachable
```

或：

```text
IPv6 = 代理侧地址
```

示例防漏配置中，Clash DNS 可设置 `ipv6: false`；如果系统网卡只有 `fe80::` 链路本地地址、没有公网 IPv6，通常不会形成公网 IPv6 泄漏。

更硬的方式是关闭 Wi-Fi IPv6：

```bash
sudo networksetup -setv6off Wi-Fi
```

恢复：

```bash
sudo networksetup -setv6automatic Wi-Fi
```
